Digital Towns Module System

What is Digital Towns?

Digital Towns is a multi-community digital marketplace platform that gives every town its own digital infrastructure. Each community gets channels, local services, buy/sell/trade groups, and a daily pulse — all running on a shared module system with per-community feature flags.

The Module System

Everything is built as a module. Each module lives in modules/[name]/ with three files: a manifest.json declaring metadata and dependencies, a migration.sql creating its database tables, and a routes.js mounting its API endpoints.

Modules are organized into layers:

  • Layer 0 — Town Infrastructure: core, auth, places, businesses, BST groups, channels, gigs, pulse, tags
  • Layer 1 — Commerce: listings, orders, payments, shipping (planned)
  • Layer 2 — Game Engine: sweepstakes, live shows, rewards (planned)
  • Layer 3 — Network: town genesis, federation, cross-town features (planned)
  • Layer 4 — Trust & Safety: moderation, reporting, content review (planned)

Sub-modules & Feature Flags

Each module can declare sub-modules — granular features that can be toggled on/off per community. For example, the auth module has sub-modules for Facebook OAuth, Google OAuth, Email OTP, Phone Verification, ID Scan, and GPS Presence. Each community's feature_flags JSONB column controls which modules and sub-modules are active.

The Tag System

Tags are the universal data layer. Content tags drive niche routing, behavioral tags are auto-applied by the system, interest tags are built from user actions, and location tags come from addresses. The tag_events table accumulates all tag interactions for analytics.

Geographic Hierarchy

The places table uses a self-referencing parent_id to model: country → state → region → town → neighborhood. Each place has its own domain, coordinates for Haversine geo-filtering, and a member count. The hierarchy is browsable at /communities.html.

Build a Module
Step-by-step guide to adding a new module

1. Create the module directory

mkdir -p modules/my-module/public

2. manifest.json

Declares metadata, layer, dependencies, and sub-modules.

{
  "id": "my-module",
  "name": "My Module",
  "version": "1.0.0",
  "layer": 0,
  "dependencies": ["core", "places"],
  "description": "What this module does.",
  "submodules": [
    {
      "id": "my-module.feature-a",
      "name": "Feature A",
      "description": "What feature A does",
      "default": true
    },
    {
      "id": "my-module.feature-b",
      "name": "Feature B",
      "description": "What feature B does",
      "default": false
    }
  ]
}

3. migration.sql

Use IF NOT EXISTS so migrations are idempotent.

-- My Module migration
CREATE TABLE IF NOT EXISTS my_things (
  id SERIAL PRIMARY KEY,
  place_id INTEGER REFERENCES places(id),
  name TEXT NOT NULL,
  data JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_my_things_place
  ON my_things(place_id);

4. routes.js

Export a function that receives app and db.

module.exports = function mountMyModule(app, db) {

  // Cookie-based auth helpers (same pattern as other modules)
  function parseCookies(req) {
    var obj = {}; var header = req.headers.cookie || "";
    header.split(";").forEach(function(p) {
      var i = p.indexOf("="); if (i < 0) return;
      obj[p.slice(0,i).trim()] = p.slice(i+1).trim();
    });
    return obj;
  }
  async function getUserId(req) {
    var sid = parseCookies(req).sid; if (!sid) return null;
    var r = await db.query(
      "SELECT user_id FROM sessions WHERE sid=$1 AND expires_at > NOW()", [sid]
    );
    return r.rows.length ? r.rows[0].user_id : null;
  }
  async function requireLogin(req, res) {
    var uid = await getUserId(req);
    if (!uid) { res.status(401).json({ error: "Login required" }); return null; }
    return uid;
  }

  // GET /api/my-module
  app.get("/api/my-module", async function(req, res) {
    try {
      var result = await db.query("SELECT * FROM my_things ORDER BY created_at DESC LIMIT 50");
      res.json({ items: result.rows });
    } catch (err) {
      res.status(500).json({ error: err.message });
    }
  });

  // POST /api/my-module
  app.post("/api/my-module", async function(req, res) {
    var uid = await requireLogin(req, res); if (!uid) return;
    try {
      var b = req.body || {};
      var result = await db.query(
        "INSERT INTO my_things (place_id, name, data) VALUES ($1,$2,$3) RETURNING *",
        [b.place_id || null, b.name || "", JSON.stringify(b.data || {})]
      );
      res.status(201).json(result.rows[0]);
    } catch (err) {
      res.status(500).json({ error: err.message });
    }
  });

};

5. Wire in server.js

Add before the view-only lockdown middleware:

try { require('./modules/my-module/routes')(app, db); }
catch(e) { console.error('my-module:', e.message); }

Update the lockdown to allow your module's paths:

if (req.path.startsWith('/api/my-module'))
  return next();

6. Update feature_flags

Add your module and sub-module flags to the community:

UPDATE communities SET feature_flags = feature_flags || '{
  "my-module": true,
  "my-module.feature-a": true,
  "my-module.feature-b": false
}'::jsonb
WHERE slug = 'digitaltowns';

Record the migration:

INSERT INTO migrations (module_name) VALUES ('my-module');
